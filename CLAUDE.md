4. **ก่อนสรุปว่า "เสร็จแล้ว" ต้อง verify จริงเสมอ** — เรียก RPC จริง / เปิดหน้าเว็บจริงดู / ล็อกอินทดสอบจริง (ใช้แถวทดสอบชั่วคราวแล้วลบทิ้ง) ห้ามเดาว่าใช้ได้จากการอ่านโค้ดอย่างเดียว
5. **6 กติกาเหล็กของสเปกตั้งต้น (server time, timezone Bangkok, กล้องสดเท่านั้น, GPS ไม่บล็อก, 1 คน/วัน, ห้าม scope creep) ยังใช้อยู่เต็มรูปแบบ**
6. **⚠️ ก่อนแก้ `dashboard.html`/`report.html`/`index.html` ทุกครั้ง ต้องเช็กก่อนว่าไฟล์ในโฟลเดอร์นี้ตรงกับที่ deploy จริงบน Netlify หรือไม่** — เคยเจอเคสที่ไฟล์ในโฟลเดอร์เก่ากว่าของจริงบนเว็บ (ฟีเจอร์รูปย่อหายไปจากไฟล์ในเครื่อง ทั้งที่ deploy ไปแล้วจริง) ถ้าแก้ทับไฟล์เก่าแล้ว deploy จะเท่ากับ**ลบฟีเจอร์ที่ใช้งานจริงอยู่ออกโดยไม่ตั้งใจ** วิธีเช็ก: `fetch(location.href).then(r=>r.text())` จากแท็บที่เปิดหน้าเว็บจริงอยู่ เทียบกับไฟล์ในเครื่อง ถ้าไม่ตรงให้ดึงของจริงมาเป็นฐานก่อนแก้ (ดูรายละเอียดเหตุการณ์ในข้อ 6.5)
7. **⚠️ ทุกครั้งที่แก้ไฟล์เสร็จ (ก่อนถามว่าจะ deploy ไหม) ต้องส่งไฟล์/ลิงก์ให้เจ้าของตรวจสอบก่อนเสมอ** (เจ้าของขอไว้ 4 ก.ค. 2569) — ใช้ `present_files` ส่งไฟล์ HTML ที่แก้ให้เจ้าของเปิดดูเองในเบราว์เซอร์ (เปิดจาก path ในเครื่อง ก็เรียก Supabase จริงได้ปกติเพราะเป็น static file + public anon key) **หมายเหตุสำคัญ:** โปรเจกต์นี้ deploy แบบ Netlify Drop (ไม่ผูก git) จึงไม่มี staging/preview URL ให้แชร์ — วิธีตรวจสอบก่อน deploy คือเปิดไฟล์ในเครื่องเท่านั้น ไม่ใช่ลิงก์เว็บจริง
8. **⚠️ ถ้ามีการแก้ไฟล์ค้างไว้ (verify ผ่านแล้วแต่ยังไม่ deploy) ต้องเตือนเจ้าของทุกครั้งที่เริ่มงานใหม่ในโปรเจกต์นี้** — เช็คได้จากหัวข้อ "อัปเดตล่าสุด" ด้านบนและข้อ 1 ว่ามีข้อความ "ยังไม่ deploy" ค้างอยู่หรือไม่ ห้ามปล่อยผ่านเงียบๆ

## 4. Supabase Edge Function

**`supervisor-photo-url`** (deploy ผ่าน Dashboard "Via Editor" ไม่ใช่ CLI)
URL: `https://aamzsbuwfdyljdvwaifb.supabase.co/functions/v1/supervisor-photo-url`

**ทำไมต้องมี:** หัวหน้าที่ล็อกอินด้วย PIN (ศุภัตรา, ผู้ช่วยแอดมิน) ไม่มี Supabase Auth session จริง จึงสร้าง Storage signed URL ตรงๆ แบบ client-side ไม่ได้ (signed URL ต้องมี auth session ตาม RLS ของ `storage.objects`)

**วิธีทำงาน:** รับ `{officer_id, pin, photo_path}` → เรียก RPC `do_supervisor_verify_pin_for_photo` ตรวจ PIN + ตรวจว่า path มีจริงในระบบ → ถ้าผ่านค่อยใช้ **service role key** (inject เป็น env var อัตโนมัติจาก Supabase ไม่ต้องพิมพ์ secret เอง) ออก signed URL อายุ 60 วิ ให้

**ตั้งค่าสำคัญ:** ปิด "Verify JWT with legacy secret" ในหน้า Settings ของฟังก์ชัน (เพราะ PIN caller ไม่มี legacy JWT) — ถ้า deploy ใหม่หรือแก้โค้ดฟังก์ชันนี้ ต้องเช็กว่า toggle นี้ยังปิดอยู่

ทดสอบแล้ว: signed URL ที่ออกมาโหลดรูปจริงได้ (200, image/jpeg), PIN ผิดถูก reject (403, `bad_pin`)

## 5. Deviations จากสเปกตั้งต้น (สิ่งที่เปลี่ยนไปจาก v1 spec เดิม)

- **3 สี → 4 สี**: เพิ่มสีส้ม "มาเลท" ระหว่างเหลืองกับแดง (เกณฑ์เวลาดูข้อ 3 แถว `06_phase_b_orange_status.sql`)
- **PIN self-service**: เดิม spec ไม่ได้ระบุรายละเอียด ตอนนี้ใช้กลไก `pin_hash IS NULL` = ต้องตั้งใหม่ ใช้ร่วมกันทั้ง "ตั้งครั้งแรก" และ "ลืมรหัสแล้วโดนแอดมินรีเซ็ต"
- **โครงสร้างหัวหน้า 3 ระดับ** (เดิม spec พูดถึงแค่ "หัวหน้า" คนเดียว): ชวนชัย (auth, สูงสุด) + ศุภัตรา (pin) + ผู้ช่วยแอดมิน (pin, ไม่ผูกชื่อคนจริงตามคำขอ) — ทั้งสามแบบตอนนี้ **เห็นข้อมูลเท่ากันหมด** รวมรูปถ่าย (ปิดช่องว่างเรื่องรูปด้วย Edge Function ในข้อ 4)
- **ชื่อเล่นในดรอปดาวน์**: เพิ่มตามคำขอ ("ยศ ชื่อ นามสกุล (ชื่อเล่น)") และเรียงตามลำดับเอกสาร PDF ต้นฉบับ ไม่ใช่ตามตัวอักษร
- **6 สีไม่ได้อยู่ในสเปกเดิม**: สเปก v1 ระบุแค่ 3 สี ตอนนี้ขยายเป็น 6: 🟢🟡🟠🔴 (สีเวลา, มีอยู่แล้ว) + 🟣 (เช็กอินไม่สมบูรณ์ — ทำแล้ว, แสดงคู่กับสีเวลา ไม่ใช่แทนที่) + 🟤 (ขาดเช็กอิน — **ยังไม่ทำ** รอระบบตารางเวรก่อน ดูข้อ 7)

## 6. บั๊กที่เจอและแก้แล้ว (กันเจอซ้ำ)

1. **`CREATE OR REPLACE FUNCTION` เปลี่ยน return columns ไม่ได้** — ต้อง `DROP FUNCTION IF EXISTS` ก่อนเสมอถ้าจะเปลี่ยนจำนวน/ชนิดคอลัมน์ที่ return (เจอซ้ำหลายรอบ: `do_check_in`, `do_list_officers`, `do_list_officers_admin`, `do_supervisor_get_today`)
2. **PostgREST embed ambiguous foreign key** — ตาราง `check_in` มี FK ไป `officer` 2 เส้น (`officer_id`, `override_by`) ถ้า query แบบ `.select("...officer(...)")` เฉยๆ จะ error "more than one relationship found" ต้องระบุ `officer!check_in_officer_id_fkey(...)` ชัดเจน
3. **RLS infinite recursion** — policy ที่ query ตารางตัวเองใน USING clause (แม้จะดูปกติ) ทำให้วนไม่รู้จบ ต้องย้าย logic ไปฟังก์ชัน SECURITY DEFINER แยก (ดูข้อ 13 ในตาราง migration)
4. **Supabase SQL Editor UI**: การพิมพ์/paste query ครั้งแรกในแท็บใหม่บางทีไม่โฟกัสจริง ทำให้ keystroke กลายเป็น keyboard shortcut ของ dashboard แทน (เช่น เผลอกด g+r แล้วเด้งไปหน้า Realtime) วิธีป้องกัน: คลิกเข้า editor แล้ว screenshot ยืนยันว่า cursor อยู่ในนั้นจริงก่อนพิมพ์ยาวๆ ทุกครั้ง
5. **⚠️ ไฟล์ในโฟลเดอร์โปรเจกต์ไม่ตรงกับที่ deploy จริง (เจอ 3 ก.ค. 2569)** — ระหว่างทำสีม่วง พบว่า `dashboard.html`/`report.html` ในโฟลเดอร์นี้เป็นเวอร์ชันเก่ากว่าที่ deploy บน Netlify จริง (ไฟล์ในเครื่องไม่มีฟีเจอร์รูปย่อ/thumbnail ทั้งที่เว็บจริงมีแล้ว) สาเหตุน่าจะเป็นแพทเทิร์นเดิมที่เคยเจอกับไฟล์ SQL migration — แก้ไฟล์ในเซสชันก่อนหน้าสำเร็จและ deploy ไปแล้ว แต่ไม่เคยเซฟ diff กลับมาไว้ในโฟลเดอร์โปรเจกต์ วิธีแก้ตอนนั้น: ดึง HTML จริงจากเว็บด้วย `fetch(location.href)` ในแท็บ Chrome ที่เปิดหน้าเว็บอยู่ (ระวัง: ผลลัพธ์จาก javascript_tool จะโดนบล็อกถ้ามี anon key/project ref อยู่ในสตริง ต้อง mask ค่าพวกนี้ก่อนส่งออกมาดู) แล้วใช้เป็นฐานแก้ต่อแทนไฟล์ในโฟลเดอร์ ดูกติกาป้องกันซ้ำในข้อ 2.6
6. **UUID ทดสอบต้องเป็น hex ที่ valid จริง** — เคยพิมพ์ UUID ทดสอบแบบ `'00000000-0000-0000-0000-00000000test'` (ใส่คำว่า "test" ต่อท้ายเพื่อให้จำง่าย) แต่ `test` ไม่ใช่ hex digit ทำให้ insert error ทันที ให้ใช้ UUID ที่เป็น hex ล้วนเสมอ เช่น `'00000000-0000-0000-0000-0000000000aa'`
7. **Supabase SQL Editor: คลิก "View full cell content" ที่ผลลัพธ์ query แล้วโฟกัสหลุดจาก editor** — หลังรัน query แล้วคลิกดูค่าเต็มในผลลัพธ์ (เช่นเพื่ออ่าน JSON ยาวๆ) ถ้า `ctrl+a` แล้วพิมพ์ทับต่อทันทีโดยไม่คลิกกลับเข้า editor ก่อน ข้อความจะไม่ถูกพิมพ์เข้า editor จริง (editor ยังเป็น query เดิม) กด Run ซ้ำจะรันคำสั่งเดิมอีกรอบโดยไม่รู้ตัว (เจอเป็น insert ซ้ำจน error duplicate key ตอนทำความสะอาดข้อมูลทดสอบ motto/สถิติ) วิธีป้องกัน: คลิกเข้าไปใน editor area ก่อนเสมอ แล้ว screenshot/get_page_text ยืนยันเนื้อหาที่พิมพ์จริงก่อนกด Run ทุกครั้งที่เพิ่งคลิกดูผลลัพธ์มา

## 20. PDPA auto photo-deletion (ลบรูปเซลฟี่อัตโนมัติ 31 วัน + retention hold) — รายละเอียด (5 ก.ค. 2569, รันอัตโนมัติผ่าน scheduled task)

**สถานะ: เขียนโค้ดเสร็จทั้ง 2 ไฟล์ (`dashboard.html`, `report.html`) + backend (migration 23) รันบน Supabase แล้วจริงและ verify ผ่านหมดด้วยข้อมูลทดสอบ (ลบทิ้งหมดหลังทดสอบแล้ว) + Edge Function `delete-old-photos` deploy แล้วจริงบน Supabase + ตั้ง pg_cron รันทุกวันแล้ว + ทดสอบ end-to-end ผ่านจริง (ทั้งเคสลบรูปและเคส retention hold รอด) + syntax check (`node --check`) ผ่านทั้ง 2 ไฟล์ + ส่งไฟล์ให้เจ้าของตรวจสอบผ่าน `present_files` แล้ว — **ยังไม่ได้ deploy ขึ้น Netlify** (เจ้าของขอให้รวม deploy กับงานข้อ 19 เป็นครั้งเดียวกัน)**

**บริบท (ที่มา):** งานนี้รันแบบอัตโนมัติผ่าน scheduled task (ไม่มีเจ้าของอยู่หน้าจอตอบคำถาม) มาจากรายการค้างใน backlog เดิม (ข้อ 7): "นโยบายลบรูปอัตโนมัติ (PDPA)" ที่ระบุไว้ตั้งแต่ v1 spec ว่ามีคอลัมน์ `photo_retention_days` รองรับแล้วแต่ยังไม่มี cron job มาลบจริง รูปเช็กอินสะสมไว้ถาวรเป็นความเสี่ยง privacy — งานนี้ทำให้ฟีเจอร์นี้เสร็จสมบูรณ์

**การตัดสินใจออกแบบ (ยืนยันไว้แล้วในเซสชันก่อนหน้า ผ่าน AskUserQuestion — scheduled task รอบนี้ไม่ถามซ้ำ):**

- **ระยะเวลาเก็บ**: 31 วัน แบบ rolling window (นับจาก `checked_in_at` ของแต่ละแถว ไม่ใช่ตามเดือนปฏิทิน) ปรับได้ภายหลังผ่าน `settings.photo_retention_days` (คอลัมน์เดิมที่มีอยู่แล้ว) — ไม่ได้รอเช็คระเบียบการเก็บข้อมูลของตำรวจ/ตม. ก่อน ตามที่ตกลงไว้
- **เก็บ metadata อื่นถาวร**: ลบแค่ตัวไฟล์รูปจาก Storage เท่านั้น ข้อมูลอื่น (สถานะ/เวลา/ระยะทาง/override/หมายเหตุ) เก็บไว้เหมือนเดิมไม่มีกำหนด

## 22. Badge ระดับทีมสำหรับกลุ่มงานธุรกิจ/ครอบครัว — ดีไซน์ยืนยันแล้ว แต่ตั้งใจ "ยังไม่เริ่มทำ" จนกว่าจะครบ ~2 สัปดาห์ (6 ก.ค. 2569)

**สถานะ: เป็นแค่การคุยออกแบบ (consultation) ไม่มีโค้ด/migration ใดๆ ในรอบนี้ — บันทึกดีไซน์ที่ยืนยันแล้วไว้เป็นแผนรอทำ ห้าม agent เริ่มเขียนโค้ดก่อนถึงกำหนดที่เจ้าของขอ (~20 ก.ค. 2569) เว้นแต่เจ้าของหยิบยกขึ้นมาเอง — รายละเอียดเต็มบันทึกไว้ใน agent memory ไฟล์ `project_team_coverage_badge_design.md` ด้วย (ไม่ใช่แค่ในเอกสารนี้)**

**คำถามเดิม (ที่มา):** ตม.จว.ระยองมี 2 กลุ่มงานที่มีเจ้าหน้าที่กลุ่มละ 2 คน คือกลุ่มงานธุรกิจและกลุ่มงานครอบครัว ทั้งสองกลุ่มมักต้องอยู่เคลียร์งาน/ทำ OT ดึกเป็นประจำ เจ้าของถามว่า ถ้ามีเจ้าหน้าที่หนึ่งคนในทีมมาพร้อมรับประชาชนก่อน 08:20 (เพราะมีคนเดียวมาพอแล้วสำหรับจุดบริการ) ควรถือว่า "ดีเยี่ยม" ทั้งคู่ไหม และอีกคนที่มักสลับกันมาสาย ควรมาไม่เกินกี่โมง

**ข้อสังเกตที่ผมท้วงติงก่อนตอบตรงๆ (Constructive Devil's Advocate):**
1. นี่คือการเปลี่ยนเกณฑ์ scoring ของสีเช็กอิน ไม่ใช่แค่ปรับ UI — สเปกตั้งต้นกำหนดเฟส D (ให้คะแนน/รางวัล) ให้รอใช้งานจริงครบ 1 เดือนก่อนตัดสินใจ ตอนนั้นใช้งานจริงมาแค่ 3 วัน (3–6 ก.ค.)
2. ถ้าให้ "ดีเยี่ยมทั้งคู่" โดยไม่มีเพดานเวลาสำหรับคนมาทีหลังเลย เท่ากับ 4 คนนี้ไม่มีความรับผิดชอบส่วนตัวเรื่องเวลาอีกต่อไป (มาคนเดียวตอน 08:00 อีกคนมาบ่ายสองก็ยังดีเยี่ยม) เสี่ยงเกิดคำถามเรื่องความเป็นธรรมจากอีก 15 คนที่ไม่มีสิทธิ์นี้

**ดีไซน์ที่ยืนยันกับเจ้าของแล้ว (ผ่าน AskUserQuestion 3 ข้อ):**
- **รูปแบบ:** แยก badge ระดับทีมต่างหาก **ไม่แตะสีรายคนเลย** (ตัวเลือกที่แนะนำ เจ้าของเลือก) — สีของแต่ละคนใน "เช็กอินวันนี้"/"ประวัติ" ยังคำนวณจากเวลาที่ตัวเองมาจริง 100% เหมือนเดิม โปร่งใส ตรวจสอบย้อนหลังได้
- **เกณฑ์ badge** (คำนวณจากเวลาเช็กอินเร็วที่สุดของสมาชิกในทีมวันนั้น):
  - ✅ "ทีมมีคนดูแลจุดบริการตรงเวลา" — มีคนมาก่อน 08:20
  - 🟡 "ทีมมีคนดูแล แต่ช้ากว่าปกติ" — ไม่มีใครมาก่อน 08:20 แต่มีคนมาก่อน 09:00 (เพดานที่เจ้าของยกตัวอย่างมาเอง — เป็นค่าตั้งแยกเฉพาะฟีเจอร์นี้ ไม่ผูกกับ green/yellow/orange_before เดิมของระบบ)
  - ⚠️ "ทีมนี้ยังไม่มีใครดูแลจุดบริการ" — เลย 09:00 แล้วยังไม่มีใครเช็กอินเลยทั้งทีม (ตรรกะคล้ายการ์ด "ขาดเช็กอิน" ที่มีอยู่แล้ว แต่มองระดับทีมแทนรายคน)
- **ข้อมูล:** เพิ่มคอลัมน์ `officer.team_name` (ธุรกิจ/ครอบครัว) พอ ไม่ต้องทำตารางทีมเต็มรูปแบบ เพราะมีแค่ 2 ทีม 4 คน — แนวทาง MVP เดียวกับระบบวันทำงาน/วันหยุด (migration 19) เวลาสร้างจริงน่าจะเพิ่ม RPC ทำนองเดียวกับ `do_get_absentees`/`do_supervisor_get_absentees` เช่น `do_get_team_coverage`/`do_supervisor_get_team_coverage`
- **จังหวะเวลา:** เจ้าของขอ **รอ ~2 สัปดาห์ก่อน** (ตกลง 6 ก.ค. 2569 → เริ่มทำได้ราวๆ 20 ก.ค. 2569) แทนที่จะรอเต็ม 1 เดือนตามกติกาเฟส D เดิม หรือเริ่มทันที — เป็นทางสายกลางที่เจ้าของเลือกเอง

**สิ่งที่ยังไม่ทำ (ตั้งใจ):**
- ยังไม่มี migration/RPC/โค้ด frontend ใดๆ สำหรับฟีเจอร์นี้ — agent ตัวถัดไปที่มารับงานต่อ **ห้ามเริ่มทำเองก่อนถึง ~20 ก.ค. 2569** เว้นแต่เจ้าของหยิบยกขึ้นมาเอง ถ้าถึงกำหนดแล้วให้เสนอเริ่มออกแบบ RPC + เพิ่มคอลัมน์ `team_name` ตามดีไซน์ข้างต้น
